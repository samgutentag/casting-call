1. What this is

casting-call takes a screen recording of a Google Meet call and produces a transcript where each line is tagged with the real name of the person who spoke it. It is a set of command-line tools you run in a terminal. There is no window, no buttons, no app to open.

The recording it expects is a stereo .mov: your microphone on the left audio channel, everyone else mixed together on the right channel, plus the Meet window visible on screen. Whisper transcribes the two audio channels into a time-stamped transcript. The catch is that the right channel mixes every remote participant into one undifferentiated voice the tool calls Caller. casting-call solves that by reading Meet's on-screen caption labels (which already say who is speaking) and using them to rewrite each Caller line with the speaker's actual name.

It is built for one person: someone who records their own calls and wants searchable, attributed transcripts of them. It runs entirely on your machine. Nothing is uploaded, there is no account, and your roster of real names is kept local and out of version control.

What it does NOT do: it does not record the call (you do that yourself with QuickTime plus a tool like Loopback), it does not do acoustic voice fingerprinting, and it does not guarantee perfect attribution when people talk over each other or when the Meet window is hidden. Anything it cannot confidently attribute stays labeled Caller rather than being guessed.

The flow is three jobs you run from the terminal: transcribe the audio (ripa), build a speaker timeline and relabel the transcript (rips), and optionally compress the video (ripv / riprec). The rest of this guide walks each one.

2. Words you need first

Recording (the .mov)
The QuickTime screen capture of your call. Two audio channels (left = you, right = everyone else) and video that shows the Meet window with captions on. Everything downstream reads from this one file.
You / Caller
The two labels Whisper assigns by channel. You is your mic (the left channel). Caller is the merged right channel, every remote person at once. casting-call's whole job is replacing Caller with real names. Lines it cannot attribute keep the Caller label.
Roster
Your local file speakers_roster.json: your own name (self) plus the people you call, each with a canonical name and optional nicknames (aliases). The tool only ever attributes a line to someone on this list. It is gitignored, so your real names stay on your machine.
Caption region
The rectangle on screen where Meet draws its caption strip, given as x,y,w,h in full-frame pixels. You find it once per recording and pass it with --region. The tool reads names only from inside this box.
Speaker timeline (speakers.json)
The intermediate file rips writes: an ordered list of spans saying "from this second to that second, the speaker was X." This is what gets matched against the transcript to relabel Caller lines.
Span
One stretch of the timeline with a single speaker: a start time, an end time, and a name (or no name, if it could not be resolved).
Unresolved / review
When the tool reads a name from the captions but cannot match it to anyone on your roster, it marks that span unresolved and asks you about it in the review step. You can name it or wave it off, in which case it falls back to Caller.
Attribution / coverage
Attribution is the act of putting a real name on a Caller line. Coverage is the score at the end: how many Caller lines got a name versus how many stayed Caller.

3. Before you start: install and roster

casting-call has no installer. You clone the repo and install its command-line dependencies through Homebrew and pip. From the README, verified against the dependency checks in bin/extract_speakers.sh and bin/extract_audio_stereo.sh:

brew install tesseract ffmpeg whisper-cpp
pip install --break-system-packages pillow numpy pytesseract pytest

Whisper also needs a model file, downloaded once. The audio scripts look for it under ~/whisper-models/ and will print this exact command if it is missing:

mkdir -p ~/whisper-models
curl -o ~/whisper-models/ggml-large-v3.bin -L \
  'https://huggingface.co/ggerganov/whisper.cpp/resolve/main/ggml-large-v3.bin?download=true'

Then create your roster from the example and edit it. Set self to your own name (this is what Meet's You caption maps to), and list everyone you call:

cp speakers_roster.example.json speakers_roster.json
# edit speakers_roster.json

A roster entry looks like this. The canonical name is what ends up in the transcript; the aliases are extra spellings or nicknames that help the fuzzy matcher when Meet shows a shortened label:

{
  "self": "Sam Gutentag",
  "members": [
    { "canonical": "Dana Park", "aliases": ["D Park", "Dana"] }
  ]
}
The tool can only attribute a line to a name that is on the roster. Someone you never added will always come back as an unresolved unknown in the review step, never as a name. Adding people you call often is the single biggest thing you can do to raise your coverage.

The tools live in bin/ as shell scripts. On Sam's machine each one is wrapped in a short shell alias (the rip* names). The alias definitions live in his separate gutils shell config, not in this repo, so if you have not set those up you can call the script directly instead (for example bash bin/extract_speakers.sh ... in place of rips ...).

5. Transcribe a recording ripa

Run ripa on a folder of recordings. With no argument it works in the current directory; the first argument is the folder and the optional second is a path to a different Whisper model.

ripa                       # transcribe every .mov in this folder
ripa ~/calls/2026-06-04    # transcribe a specific folder

It works in two passes. First it extracts a stereo MP3 from every .mov into an audio_only/ folder. Then it splits each recording into two mono tracks (your channel and the caller channel), runs Whisper on each, and merges the two results into one transcript sorted by time, with every line tagged [You] or [Caller]. Output lands in transcripts/. Files that already have a transcript are skipped, so re-running is safe and cheap.

One important behavior: at the very end of each file, if a speakers.json already sits next to the recording, ripa automatically relabels the Caller lines from it and prints a coverage line. That means once you have run rips (next section) to produce the timeline, transcription and attribution happen together. If there is no speakers.json, you simply get plain [Caller] lines and nothing breaks.

A finished transcript line looks like this:

[0:04:12] [You] So where did we land on the rollout?
[0:04:18] [Caller] We pushed it to next sprint.

6. Attribute speakers rips

This is the core of the tool. rips takes a recording and the caption region, reads the captions frame by frame, and writes a speakers.json timeline next to the recording. Add --transcript and it will also rewrite that transcript's Caller lines in place.

# build the timeline only
rips recording.mov --region 120,1850,900,180

# build the timeline AND relabel an existing transcript in place
rips recording.mov --region 120,1850,900,180 --transcript transcripts/recording.txt

Every flag, read from casting_call/cli.py:

--region x,y,w,hyes in v1The caption strip rectangle in full-frame pixels. Required: if you omit it the tool exits with an error. Auto-locating it is a planned follow-up, not built yet.
--roster PATHnoPath to the roster file. Defaults to speakers_roster.json at the repo root.
--out DIRnoWhere to write speakers.json, the extracted frames/, and the review/ crops. Defaults to the same folder as the recording.
--transcript PATHnoAn existing transcript to relabel in place. Without it, rips only writes the timeline.
--callers NnoHow many OTHER people are on the call. 1 forces every Caller line onto the single other speaker; 2 or more forces timed, per-line attribution. Omit it to auto-detect.

7. Finding the caption region

The hardest part of using rips is the --region value, because the Meet window can sit anywhere on screen and its position is not guaranteed to be the same from one recording to the next. You give the tool a rectangle so it knows where to look for caption text and ignores the rest of the screen.

The format is four numbers, comma-separated, in full-frame pixels of the original recording (which can be Retina-resolution, for example 4096 by 2304): x,y,w,h. x,y is the top-left corner of the box, w,h is its width and height. Aim the box at the bottom strip of the Meet window where captions appear, generous enough to catch the name labels but tight enough to exclude other on-screen text.

The practical way to find it: open one frame of your recording in an image editor or Preview, draw a selection over the caption strip, and read off the pixel coordinates. You only need to do this once per recording. If the box is wrong, the tool will either read nothing (no signal, everything stays Caller) or read the wrong region.

8. The review step

When rips reads a name from the captions that it cannot match to anyone on your roster, it does not guess. It marks that span as an unresolved unknown and, after building the timeline, walks you through them in the terminal. It tells you up front how many distinct unknowns there are, so you can decide whether it is worth the effort.

For each unknown it saves a cropped image of the caption (named unknown_01.png, unknown_02.png, and so on) into a review/ folder, shows you the raw OCR text it read, and prompts you for a name:

3 distinct unknown speaker(s) to review.
Enter a name for each, or "skip"/Enter to make the rest Caller.

unknown_01 (raw OCR "Jrdn M.", crop unknown_01.png) ->

Open the crop image to see who it is, type their name, and press Enter. You can stop at any time: typing skip or just pressing Enter ends review immediately and collapses every remaining unknown to Caller. You are never forced to clear the whole queue. Anything you do not resolve simply stays Caller, which is the same as the old behavior, so review can only improve the result, never make it worse.

If you find yourself naming the same person across many calls, add them to speakers_roster.json. Once they are on the roster the tool matches them automatically and they stop showing up in review.

9. 2-person vs group calls

The --callers flag controls how Caller lines get attributed, and it matters because the two cases are genuinely different.

On a 2-person call there is only one other voice, so timing does not matter: every Caller line must be that one person. The tool detects this automatically when the timeline contains exactly one named non-self speaker and collapses all Caller lines onto them. You will see a message like 2-person call: 'Dana Park' is the only other speaker -> all Caller lines attributed to them.

On a group call, timing is everything: each Caller line is matched to whoever the timeline says was speaking at that moment. To force this, pass --callers 2 (or higher). To force the single-speaker collapse instead, pass --callers 1. If you pass --callers 1 but the tool actually detected more than one speaker, it will warn you and fall back to timed attribution rather than mislabel everyone.

Verified against casting_call/transcript.py: a timed relabel looks up the speaker at each line's timestamp shifted back by the caption-lag offset (1.5 seconds by default), because captions trail the spoken audio. Spans where the timeline says you were speaking are treated as bleed from your mic into the caller channel and stay Caller.

10. Compress the video ripv riprec

These are housekeeping tools, separate from transcription. ripv takes a folder and compresses each .mov into a smaller 1080p MP4 under converted_video/, useful for archiving raw recordings that are large. riprec is a convenience wrapper that runs audio extraction and video compression together.

ripv ~/calls/2026-06-04     # compress every .mov in the folder to mp4
riprec ~/calls/2026-06-04   # extract audio + compress video in one pass

Neither of these does speaker attribution. They exist because the same recordings that you transcribe are also the ones you want to shrink for storage.

11. Where the data comes from

It is worth being precise about which file supplies what, because the words and the names come from two completely separate sources.

The transcript text comes only from the audio, through Whisper. The left and right channels are split, transcribed separately, and merged by timestamp. The on-screen captions are never used for the words themselves, because caption text is too low-quality. The captions supply only the speaker names.

The speaker names come from the video: rips samples frames at 2 frames per second, crops the caption region, OCRs it with tesseract, and fuzzy-matches the leading name token against your roster. A match above the threshold (0.72 by default) becomes that person's canonical name; You maps to your self name; anything below threshold becomes an unknown for review.

A few defaults shape the result, all read from casting_call/config.py:

SettingDefaultWhat it controls
frame rate2 fpsHow often frames are sampled. Speaking turns run longer than this, so 2 fps is enough.
match threshold0.72How close an OCR'd name must be to a roster name to count as a match. Below it, the name goes to review.
debounce1.0 sSpeaker flickers shorter than this are dropped, so a one-frame misread does not fragment a turn.
grace bridge3.0 sA short gap flanked by the same speaker is absorbed, so a quick window switch does not split one turn into Name / Caller / Name.
caption lag1.5 sCaptions trail the audio, so timed lookups are shifted back by this much to line up with Whisper's timestamps.
These are defined in code, not exposed as command-line flags. To change them you edit config.py. The only knobs on the command line are --region, --roster, --out, --transcript, and --callers.

12. Good to know

Does it send my calls or names anywhere?
No. Everything runs locally: ffmpeg, Whisper, and tesseract all run on your machine, and there is no account or network upload. Your roster of real names is gitignored and stays on disk.
What if the captions were off during the call?
Then there is nothing for rips to read, no speakers.json gets a useful timeline, and every line stays Caller. Captions must be ON during the recording. This is the one setup requirement that the tool cannot work around.
What if the Meet window was hidden or I switched tabs mid-call?
Those frames fail an occlusion check (the caption strip is normally a dark translucent band; when Meet is covered the region reads as something else) and are marked "no signal." Attribution falls back to Caller for that stretch, but the words are never lost, because the audio is recorded independently of what is on screen.
It attributed a line to the wrong person. Why?
Most often the caption lag or a turn boundary: a line right at the handoff between two speakers can land on the previous one. The 1.5 s lag offset corrects the common case but not every edge. Heavy crosstalk is explicitly not solved; ambiguous lines are meant to stay Caller rather than be guessed.
Can I just run it again if I get it wrong?
Yes. rips rewrites speakers.json each run, and --transcript relabels in place. Re-run with a corrected --region or an updated roster. Note that relabeling is in place, so keep an original copy of a transcript if you want to compare.
What does the coverage number mean?
It is the share of originally-Caller lines that ended up with a real name. "82%" means 18% of caller lines stayed Caller (captions off, window covered, below-threshold names you skipped, or genuine silence). It is a quick read on how well attribution went.
Do I have to use the rip aliases?
No. They are personal shell shortcuts defined outside this repo. You can call the scripts directly, for example bash bin/extract_speakers.sh recording.mov --region ... from the repo root.
